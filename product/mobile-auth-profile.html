Авторизация и профиль

Сценарий для mobile-клиента эксперта: вход по PIN, обновление access token и проверка текущей сессии.

Что делает сценарий

1. Получает access token и refresh token через POST /api/v1/auth/token.

2. Обновляет access token без повторного ввода PIN.

3. Проверяет текущего пользователя через GET /api/v1/users/me.

Prerequisites

Нужны PIN эксперта, client_type=native и HTTPS-доступ к https://api.lgtime.ru/api/v1.

Шаги

1. Отправить форму на POST /auth/token с grant_type=expert_pin.

2. Сохранить access_token, expires_in и refresh_token.

3. Перед рабочими запросами вызывать GET /users/me.

4. Когда access token истёк, повторить POST /auth/token с grant_type=refresh_token.

Endpoints

POST /api/v1/auth/token

grant_type=expert_pin, client_type=native, pin, опционально username и device_id.

POST /api/v1/auth/token

grant_type=refresh_token, client_type=native, refresh_token.

GET /api/v1/users/me

Требует заголовок Authorization: Bearer <access_token>.

Request

curl -sS -X POST 'https://api.lgtime.ru/api/v1/auth/token' \ -H 'Content-Type: application/x-www-form-urlencoded' \ --data 'grant_type=expert_pin&client_type=native&pin=1234'
curl -sS -X POST 'https://api.lgtime.ru/api/v1/auth/token' \ -H 'Content-Type: application/x-www-form-urlencoded' \ --data 'grant_type=refresh_token&client_type=native&refresh_token=<refresh_token>'
curl -sS 'https://api.lgtime.ru/api/v1/users/me' \ -H 'Authorization: Bearer <access_token>'

Response

{ "access_token": "<jwt>", "token_type": "Bearer", "expires_in": 3600, "refresh_token": "<refresh>" }
{ "id": 7, "email": "expert@example.com", "organization": { "id": 1, "name": "ActPro", "office_email": "office@example.com" }, "roles": ["expert"] }

Ошибки

400 при неправильном grant_type или пустых обязательных полях.

401 при битом или просроченном access token на GET /users/me.

429 при rate limit на PIN-вход.

Edge cases

Если передан username, он должен совпадать с владельцем PIN.

Для client_type=native refresh token надо передавать в теле формы, не в cookie.

Что хранить локально

access_token, refresh_token, время истечения access token и при необходимости кэш ответа /users/me.

Текущие ограничения

GET /users/me возвращает только id, email, organization и roles. Полного mobile profile endpoint сейчас нет.
