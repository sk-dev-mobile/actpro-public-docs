Фото и отправка акта

Сценарий offline-friendly загрузки временных фото и финальной отправки акта одним JSON payload.

Что делает сценарий

1. Загружает фотографии по одной через POST /api/v1/files/photo.

2. Хранит локальную связку между фото, client_photo_key и полученным file_id.

3. Финализирует акт через POST /api/v1/acts.

Prerequisites

Нужны access token эксперта, заранее загруженный каталог и локальный UUID черновика draft_key.

Шаги

1. Для каждого фото отправить multipart upload с draft_key и уникальным client_photo_key.

2. После успешного ответа сохранить file_id.

3. Собрать финальный JSON с client_act_key, client_id, данными объекта и дефектами.

4. Отправить POST /api/v1/acts и при сетевом retry повторять тот же JSON без изменений.

Endpoints

POST /api/v1/files/photo

multipart/form-data: draft_key, client_photo_key, file.

POST /api/v1/acts

application/json: финальный payload акта с дефектами и массивами file_ids.

Request

curl -sS -X POST 'https://api.lgtime.ru/api/v1/files/photo' \ -H 'Authorization: Bearer <access_token>' \ -F 'draft_key=2d78a678-9d4a-4f43-9a3e-602ca5fcdd4d' \ -F 'client_photo_key=2ad8412f-2ab9-411b-92d4-b3ec0221e605' \ -F 'file=@photo.jpg;type=image/jpeg'

{ "act_id": 42, "status": "finalized", "created_at": "2026-04-16T18:00:03Z", "client_act_key": "c8fbc4b8-4d2d-4c8e-b79c-0df508db6b0a", "act_type": "expertise", "schema_version": "v1", "attached_files_count": 2 }

Ошибки

400 и 422 для битой формы upload или битого JSON submit.

401 и 403 для невалидной сессии или неэкспертной роли.

409 ACT_IDEMPOTENCY_CONFLICT если тот же client_act_key отправлен с другим payload.

409 ACT_FILES_CONFLICT если хотя бы один file_id не принадлежит организации или не находится в статусе uploaded_temp.

Edge cases

Повторный upload с теми же draft_key и client_photo_key допустим только для того же файла. Иной контент требует новый client_photo_key.

На момент сабмита каждый file_id должен быть в статусе uploaded_temp; после успеха сервер переводит его в attached.

inspection_finished_at не может быть раньше inspection_started_at.

Что хранить локально

draft_key, список локальных фото с их client_photo_key и полученными file_id, а также неизменяемый client_act_key для retry.

Текущие ограничения

Upload принимает только image/jpeg и image/png до 20 МиБ на файл. Временные файлы хранятся 72 часа от последней успешной загрузки или идемпотентного повтора.
